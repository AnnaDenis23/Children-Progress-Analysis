# Children Progress Analysis

## Описание проекта

Проект посвящён анализу данных по занятиям детей с целью выявления случаев отсутствия прогресса (stagnation) и зон риска в развитии.

Основная задача — автоматически определить детей, у которых:

* длительное время не наблюдается улучшение результатов
* есть риск застоя в развитии
* наблюдается слабая динамика после длительного плато

Проект имитирует реальную задачу аналитика: работа с "грязными" данными, построение логики анализа и формирование отчётов.

---

## Данные

Входной файл:

```
data/children_sessions.xlsx
```

Содержит информацию о занятиях:

* `child_id` — идентификатор ребенка
* `age` — возраст
* `diagnosis` — диагноз
* `domain` — навык (например, Listening, Social)
* `session_date` — дата занятия
* `assessment_score` — оценка прогресса
* `comment` — комментарий специалиста
* `progress_flag` — исходный флаг прогресса (часто некорректный)
* `specialist_type` — тип специалиста

---

## Проблемы в данных

В исходных данных были обнаружены ошибки:

* `progress_flag` иногда содержит тип специалиста
* `specialist_type` часто пустой
* значения могут быть на русском (`импровед`, `стагнант`)
* присутствуют пропуски и некорректные значения

В проекте реализована автоматическая очистка:

* исправление опечаток (`импровед` → `improved`)
* перенос значений в правильные колонки
* заполнение пропусков (`unknown`)

## Архитектура решения

Проект реализован как простой аналитический пайплайн:

1. Загрузка данных из Excel
2. Очистка и валидация (`validation.py`)
3. Расчёт метрик и логики stagnation (`analysis.py`)

Такое разделение упрощает поддержку, тестирование и расширение проекта.

## Логика анализа

### 1. Автоматический progress_flag

Исходный `progress_flag` ненадежен, поэтому рассчитывается заново:

* score вырос → `improved`
* score не изменился или снизился → `stagnant`
* первая запись → `unknown`

---

### 2. Основное правило stagnation

Для каждой пары `child_id + domain`:

* определяется дата последнего улучшения
* рассчитывается количество дней без прогресса

Если:

```
days_without_progress >= 28
```

→ случай считается stagnation

---

### 3. Анализ последних 28 дней

Дополнительно анализируются последние 28 дней:

* количество занятий
* значения score
* наличие прогресса

Добавленные поля:

* `sessions_last_28_days`
* `scores_last_28_days`
* `progress_in_last_28_days`

---

### 4. Выявление плато (plateau)

Если:

* длительное время не было прогресса
* и только в конце появился слабый рост

→ случай помечается как:

```
plateau_risk
```

## Компромиссы

В проекте приняты упрощения:

* progress_flag считается только по score
* не используется анализ текстов (NLP)
* фиксированный порог stagnation (28 дней)
* не учитывается частота занятий

Это позволяет сохранить простоту и интерпретируемость модели.

---

## Ограничения

* не учитывается сложность заданий
* интервалы между занятиями могут искажать анализ
* отсутствует анализ контекста комментариев
